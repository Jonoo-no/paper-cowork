# 论文流水线

你正在运行 paper-cowork 插件的论文写作核心流程。每次运行从 `.pipeline-state.json` 读取进度，从断点继续执行。

**严格遵守以下规则：**
- 每个 Phase 结束后更新 `.pipeline-state.json`，问用户是否继续下一 Phase
- 参考文献**仅从用户文献笔记中提取**，不凭空编造任何引用
- 实验数据**仅从用户提供的素材中引用**，缺失部分标注 `⚠️ 待补充`
- 所有文件写入使用 `workspacePath` 为根目录

---

## 0. 前置检查

开始前执行：

1. 检查 `{workspacePath}/论文项目/毕业论文/.pipeline-state.json` 是否存在
   - **存在** → 读取 state，输出当前进度摘要
   - **不存在** → 输出提示「未检测到工作区，请先运行 /启动论文工作区」

2. 输出当前进度，例如：
   ```
   📍 当前进度
   专业：计算机科学
   论文大类：理工科
   论文类型：实践型
   已完成：论文主题确认
   当前阶段：文献调研（Phase 2）
   ```

3. 问：「要继续吗？按 Enter 继续，输入 '退出' 结束。」

---

## Phase 1: 论文主题确认

当 `state.currentPhase = 0 或 1` 时执行。

① 问用户以下问题（逐一提问，每次等回答）：

   **「1. 你的专业是什么？」**
   用户回答后，映射到 4 个大类：

   | 大类 | 包含专业 |
   |------|---------|
   | **理工科** | 计算机、电子、机械、土木、自动化、通信等 |
   | **理科 / 医学** | 数学、物理、化学、生物、临床医学、药学等 |
   | **文科** | 中文、历史、哲学、外语、新闻传播等 |
   | **经管 / 社科** | 经济学、管理学、法学、教育学、社会学等 |

   **「2. 论文类型是什么？」**
   - **综述型**：以文献分析为主，不需要实验
   - **实践型**：有实验/设计/开发/调研工作（理工科、医学）
   - **实证型**：有数据分析和模型验证（经管社科）

   **「3. 学校对字数有要求吗？」**（如 8000 字、10000 字等）

   **「4. 有没有初步的选题方向或想法？」**

② 根据用户回答，对话式地收敛题目。给出 1-2 个候选题目，让用户选择或修改。

③ 写入 `{workspacePath}/论文项目/毕业论文/01-论文蓝图.md`：

   ```markdown
   # 论文蓝图

   - **专业**：[用户回答]
   - **论文大类**：[映射的大类]
   - **论文类型**：[综述型/实践型/实证型]
   - **字数要求**：[字数]

   ## 题目

   [确定的题目]

   ## 研究方向

   [简要描述]

   ## 预期章节结构

   根据论文大类生成默认结构，允许用户修改：

   ### 理工科
   1. 绪论（背景、意义、国内外现状、论文结构）
   2. 相关技术介绍
   3. 系统设计与实现
   4. 实验与分析
   5. 总结与展望

   ### 理科 / 医学
   1. 引言（背景、文献综述、研究目的）
   2. 材料与方法
   3. 结果
   4. 讨论
   5. 结论

   ### 文科
   1. 绪论
   2. 理论基础与概念界定
   3. [主体分析章节]
   4. [主体分析章节]
   5. 问题与建议 / 结语

   ### 经管 / 社科
   1. 绪论
   2. 理论基础与文献综述
   3. 现状分析
   4. 实证分析 / 案例分析
   5. 结论与建议

   ## 需要关注的文献方向
   - [ ] ...
   ```

   生成后让用户确认章节结构，用户可自由增删章节。

④ 更新 `.pipeline-state.json`：

```json
{
  ...现有字段,
  "major": "计算机科学",
  "paperCategory": "理工科",
  "paperType": "实践型",
  "wordCount": 10000,
  "title": "确认的题目",
  "chapters": ["绪论", "相关技术", "系统设计", "实验", "总结"],
  "currentPhase": 2
}
```

⑤ 问：「论文蓝图已生成。是否进入下一阶段——文献调研？」

## Phase 2: 文献调研

当 `state.currentPhase = 2` 时执行。

① 检查 `{workspacePath}/文献笔记/` 目录：

   列出已有文献笔记（`.md` 文件），统计数量。
   - 如果少于 3 篇：提醒「文献笔记数量较少，建议至少阅读 10-15 篇核心文献后再开始写作」

② 生成检索关键词：
   - 根据论文题目和方向，生成中英文检索关键词
   - 告知用户：「你可以用以下关键词去知网、万方、Google Scholar 检索文献」

③ 中文文献引用来源说明：
   - 告知：「中文文献请从知网（cnki.net）检索后导出」
   - 导出格式指导：「选中文献 → 导出/参考文献 → 选择 EndNote 格式 → 下载 .ris 文件」
   - 「把文献信息整理到 Obsidian 文献笔记模板中放入 文献笔记/ 目录」

④ 英文文献补充（可选）：

   检查 `nature-academic-search` 是否可用（查看 `.claude/agents/nature-academic-search.md` 是否存在）：
   - **可用**：自动搜索 3-5 篇相关英文文献，输出摘要作为补充参考
   - **不可用**：跳过，告知用户可手动搜索

⑤ 读取已有文献笔记：

   读取 `文献笔记/` 下所有 `.md` 文件，提取每篇的：
   - 核心贡献
   - 可引用观点
   - 与本研究的关系

   按主题分类整理，写入 `{workspacePath}/论文项目/毕业论文/02-文献综述骨架.md`：

   ```markdown
   # 文献综述骨架

   ## 一、文献分类

   ### 主题 1：[主题名]
   - [文献A] → 核心观点
   - [文献B] → 核心观点

   ### 主题 2：[主题名]
   ...

   ## 二、研究空白（Gap）分析

   已有研究主要集中在 xxx，但在 xxx 方面存在不足...

   ## 三、与本论文的关系

   - 理论基础：xxx
   - 方法参考：xxx
   - 对比基线：xxx
   ```

⑥ 问：「文献综述骨架已生成。是否进入下一阶段——参考论文分析？」

   更新 state：`currentPhase = 3`

## Phase 3: 参考论文分析

当 `state.currentPhase = 3` 时执行。

① 检查 `{workspacePath}/参考论文案例/` 目录：

   列出已有文件：
   - 如果有用户自己写的分析笔记（`.md`）→ 读取内容，纳入后续写作参考
   - 如果只有模板文件或为空 → 引导用户放入参考论文

② 如果目录为空，问：
   **「参考论文案例目录没有内容。你有想分析的参考论文吗？如果有，请放入该目录（放 PDF 或你的分析笔记），然后告诉我继续。」**
   - 用户放入了 → 读取并分析
   - 用户说「不需要」→ 跳过本阶段

③ 分析每篇参考论文（已有分析笔记的直接引用；PDF 文件读取摘要/引言部分）：

   按以下维度分析，输出到 `{workspacePath}/参考论文案例/{论文名}-分析.md`：

   ```markdown
   # 参考论文分析：[标题]

   ## 结构分析
   - 章节组织方式
   - 引言结构类型（漏斗式/背景→gap→本文）
   - 论证展开逻辑

   ## 段落技巧
   - 主题句写法
   - 段落衔接方式
   - 节奏感

   ## 论证方式
   - 引用插入模式
   - 证据展示顺序
   - 讨论方法

   ## 可借鉴点
   - 本篇论文可借鉴的写作技巧
   ```

④ 汇总所有参考论文的写作技巧，写入 `{workspacePath}/参考论文案例/写作技巧汇总.md`。

⑤ 问：「参考论文分析完成。是否进入下一阶段——素材收集？」

   更新 state：`currentPhase = 4`

## Phase 4: 数据与素材收集

当 `state.currentPhase = 4` 时执行。

① 根据论文大类，逐项询问用户：

   **理工科实践型：**
   - 系统架构图 / 流程图 有吗？
   - 核心代码或算法伪代码？
   - 实验环境（硬件、软件版本）？
   - 实验结果数据？
   - 对比实验设置与结果？

   **理科 / 医学实践型：**
   - 实验材料 / 试剂 / 样本信息？
   - 实验方法与步骤？
   - 原始数据（统计表格、图表）？
   - 统计分析结果（p 值、置信区间等）？

   **文科综述型：**
   - 分析框架或理论依据？
   - 对比分析维度？
   - 文本案例或历史资料？

   **经管社科实证型：**
   - 数据来源（问卷、数据库、统计年鉴）？
   - 变量定义与测量方式？
   - 模型设定？
   - 回归 / 统计分析结果？

   每问一项都等用户回答，用户可以：
   - 「有，已放入素材/目录」→ 检测文件存在
   - 「还没有」→ 标注 ⚠️ 待补充
   - 「这个不需要」→ 跳过

② 更新 `03-素材清单.md`：

   ```markdown
   # 素材清单

   ## 已有材料
   | 素材名 | 路径 | 状态 | 对应章节 |
   |--------|------|------|---------|
   | 实验结果表 | 素材/实验数据/result.csv | ✅ | 实验与分析 |

   ## 缺失材料
   | 需要什么 | 对应章节 | 说明 |
   |---------|---------|------|
   | 系统架构图 | 系统设计 | ⚠️ 待补充 |
   ```

③ 问：「素材清单已整理。是否进入下一阶段——论文写作？（这是最长的阶段）」

   更新 state：`currentPhase = 5`

## Phase 5: 论文写作

当 `state.currentPhase = 5` 时执行。这是最长的阶段，逐章迭代。

① 确定章程序列：

   从 state 中读取 `chapters` 列表。Abstract 和 Title 在最后处理。

   示例顺序（理工科）：绪论 → 相关技术 → 系统设计 → 实验 → 总结 → Abstract → Title

② 逐章推进（循环）：

   对未完成的章节执行以下操作：

   ### 5a. 起草初稿

   根据以下资料生成初稿：
   - `01-论文蓝图.md` — 论文方向与结构
   - `02-文献综述骨架.md` — 已有文献及引用
   - `03-素材清单.md` — 已有数据
   - `参考论文案例/写作技巧汇总.md` — 参考写作技巧
   - 已完成章节的内容（保持风格一致）

   **引用规则：**
   - 仅引用 `文献笔记/` 中存在的文献
   - 格式：在文字后标注 `[作者，年份]`，如「张三等人（2024）提出了...」
   - 如果某个论点需要引用但笔记中没有 → 标注 `⚠️ 建议补充文献`

   **数据规则：**
   - 仅使用 `03-素材清单.md` 中标记 ✅ 的数据
   - 缺失数据标注 `⚠️ 待补充`

   **写作风格：**
   - 中文写作，学术语体
   - 不堆砌形容词，不使用「首个」「首次」「前所未有的」等夸张表述
   - 段落有主题句，段间有过渡

   完成后问用户：「{章节名} 初稿已生成。请审阅并给出修改意见。」

   ### 5b. 用户审阅 + 修改

   - 用户给出意见后，按意见修改
   - 最多迭代 3 轮
   - 用户确认「通过」后进入下一章

   ### 5c. 写入文件

   写入 `{workspacePath}/论文项目/毕业论文/05-初稿/0N-章节名.md`

   每完成一章，更新 state 中记录已完成章节。

③ 所有章节完成后，基于全文写 **Abstract**。

④ 基于全文构思 **Title**，给 3-5 个候选让用户选。

⑤ 全部完成后：

   更新 state：`currentPhase = 6`, `draftComplete = true`

   问：「论文初稿已完成！是否进入下一阶段——格式输出、引用整理与降重？」

---

## Phase 6: 格式 + 引用 + 降重

当 `state.currentPhase = 6` 时执行。

### 6a. 引用整理

1. 扫描 `文献笔记/` 下所有 `.md` 文件，提取 frontmatter 中的文献元数据
2. 汇总为参考文献列表
3. 检查 `scripts/docx_converter.py` 是否存在：

   - **存在且 Python 可用**：
     把文献元数据整理为 JSON 格式，提示用户在 Phase 6e 输出时使用脚本生成

   - **不存在或无 Python**：
     AI 直接生成 GB/T 7714 格式参考文献列表，但告知用户：
     「⚠️ 自动生成的参考文献格式可能不精确，建议手动核对」

   GB/T 7714 示例格式：
   ```
   [1] 作者1, 作者2. 论文标题[J]. 期刊名, 年份, 卷(期): 起止页码.
   [2] 作者. 专著书名[M]. 出版地: 出版社, 年份.
   [3] 作者. 论文标题[C]//会议论文集名. 会议地点, 年份: 起止页码.
   ```

4. 写入 `{workspacePath}/论文项目/毕业论文/参考文献列表.md`

### 6b. 图表自查

遍历 05-初稿/ 下所有章节：
- 检查图编号（图 1、图 2...）是否连续
- 检查表编号（表 1、表 2...）是否连续
- 检查每个图/表是否有正文引用（"如图 X 所示""见表 X"）
- 输出自查报告

### 6c. 内容自查

- 检查是否有残留的 `⚠️ 待补充` 标记
- 检查是否有 `⚠️ 建议补充文献` 标记
- 检查数字格式、单位格式是否统一
- 输出自查报告

### 6d. 降重（可选）

问：「论文是否需要降重？推荐中等程度。」

提供 4 个选项：
1. **不需要** → 跳过
2. **轻度降重** — 主要替换 AI 高频词
3. **✅ 中等降重（推荐）** — 词级替换 + 句级重构
4. **重度降重** — 句级重构 + 段落调整 + 三轮迭代

如果用户选择降重：
- 检查 `aigc-reduce` 是否可用（查看 `.claude/agents/` 或 `.claude/skills/aigc-reduce/`）
- 可用 → 调用 `/aigc-reduce` skill 执行降重
- 不可用 → 告知：「aigc-reduce 未安装。你可以手动从 [...] 安装，或我按降重原则手动修改」

### 6e. 输出 .docx

1. 检查 `{workspacePath}/论文模板/` 下是否有 `.docx` 模板

2. 检查 `{workspacePath}/../scripts/docx_converter.py` 是否存在 + Python 是否可用：

   **Python + 脚本 + 模板 全齐：**
   ```bash
   python {scripts_path}/docx_converter.py \
     --template {workspacePath}/论文模板/学校模板.docx \
     --chapters {workspacePath}/论文项目/毕业论文/05-初稿/ \
     --references {workspacePath}/论文项目/毕业论文/参考文献.md \
     --output {workspacePath}/论文项目/毕业论文/论文终稿.docx
   ```
   告知用户运行以上命令。

   **缺 Python 或脚本：**
   - 输出合并后的完整 Markdown 文件：`论文终稿.md`
   - 告知：「你的电脑未检测到 Python/python-docx，无法自动生成格式化 .docx。」
   - 方案 A：运行 `pip install python-docx` 安装后重试
   - 方案 B：用 pandoc 转换：`pandoc 论文终稿.md --reference-doc=学校模板.docx -o 论文终稿.docx`
   - 方案 C：用 Word 打开 .md 文件手动套模板

3. 输出终稿路径给用户

更新 state：`currentPhase = 7`

问：「论文终稿已生成！是否进入下一阶段——修改追踪？如果暂时不需要，以后随时可以再运行 /论文流水线 进入此阶段。」

## Phase 7: 修改追踪

当 `state.currentPhase = 7` 时执行。

① 问：「是收到导师反馈了吗？请把导师的意见贴给我，或者描述修改要求。」

② 逐条分析意见，生成修改方案：

   ```
   ## 修改方案

   | 导师意见 | 修改方式 | 涉及文件 |
   |---------|---------|---------|
   | 第三部分不够详细 | 补充 xxx 内容 | 05-初稿/03-系统设计.md |
   | 缺少对比实验 | ⚠️ 需要补充数据 | 05-初稿/04-实验.md |
   ```

   对每条意见标注：
   - AI 可自行修改的（文字润色、结构调整）
   - 需要用户补充材料的（数据、引用等）

③ 用户确认方案后执行修改

④ 版本归档：

   ```
   {workspacePath}/论文项目/毕业论文/06-修改稿/
   └── v{版本号}/
       ├── 各章节.md
       └── 修改说明.md
   ```

   版本号从 v2 开始（v1 是初稿）。

   `修改说明.md` 格式：

   ```markdown
   # 修改说明 v{版本号}

   修改日期：YYYY-MM-DD

   | 意见 | 修改内容 | 状态 |
   |------|---------|------|
   | xxx  | xxx     | ✅  |
   ```

⑤ 更新 state：`version` 递增

⑥ 问：「修改完成。还有其他要修改的吗？」

   - 有 → 回到 ①
   - 没有 → 结束流程

---

## Phase 完成后的通用收尾

每个 Phase 完成后：

1. 更新 `.pipeline-state.json` 中的 `currentPhase` 和进度记录
2. 问：「是否进入下一阶段？（输入 '继续' 或按 Enter）」
3. 用户输入 `退出`、`下次` → 保存 state，结束：「已保存进度。下次运行 /论文流水线 会从这里继续。」

## 设计决策

- state 文件是 JSON 格式，手动可读可修改
- 所有引用必须来自用户的文献笔记——AI 不生成虚构引用
- 数据缺失标注 ⚠️ 而不是 AI 编造
- 每阶段用户确认后才推进——用户始终控制节奏
- 章节结构根据论文大类自动适配，但用户可以自由修改
- 降重默认推荐中等程度，尊重用户选择
- 文档生成优先走 python-docx 脚本，无 Python 则降级为 markdown 输出
